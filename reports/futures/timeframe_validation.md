## Confirmed Timeframe for All Backtests

Every backtest run in all three years used **1h (one-hour) bars** as the trading timeframe.  
This was confirmed by cross-checking three independent sources:

| Source | Evidence |
|---|---|
| Log output | `MES: 1m data not available — using 1h for research` printed for every symbol |
| Code path | `_best_available_timeframe(symbol, data_dir, "1m")` returned `"1h"` — no `MES_1m.csv` exists |
| Data inventory | `data/futures/historical/` contains `MES_1h.csv` (11,227 bars), no `MES_1m.csv` |

The HTF (higher-timeframe) data used was **4h bars**, generated by resampling the 1h Yahoo download.

## Why 2024 Backtest Only Covers ~Half the Year

The 1h download was capped at 720 days back from 2026-05-31, giving a start date of approximately 2024-06-10. The `run_backtest_year` function requests `datetime(2024, 1, 1)` but bars before ~June 2024 simply do not exist in the CSV. The engine loads 0 bars for Jan–May 2024 and starts trading from the first available bar.

**Practical effect:** The 2024 results represent approximately 7 months (Jun–Dec), not a full year. Trade count is lower than 2025 as a result (211 MES vs 388 MES). The 2024 numbers are not comparable to 2025 on an annualised basis.

---

## Labeling Defect in Published Reports

The JSON summary files (`reports/futures/research/research_summary_*.json`) store `"symbol": "MES"` without a timeframe field. The gate therefore reports `"unknown"` in the TF column.

The `PerformanceReport.symbol` field was correctly set to `"MES[1h]"` inside the backtest engine, but `build_summary_table` uses the outer dict key (`"MES"`) not `report.symbol`. This is cosmetic — the timeframe actually used is 1h as confirmed above.

**Fix:** Add `"timeframe_used": trading_tf` to each row dict in `build_summary_table()`.

## What 1h Bars Mean for This Strategy

The ICT Silver Bullet is designed for **1-minute bars**. On 1h bars:

| Parameter | On 1m bars (intended) | On 1h bars (used) |
|---|---|---|
| Session window bars | ~60–120 per session | 1–3 per session |
| FVG detection sensitivity | Fine-grained, many signals | Coarse, fewer signals |
| SL accuracy vs sweep wick | Within 5–15 pts | Within 50–200 pts |
| Entry precision | Tick-level limit orders | 1h OHLC approximation |
| Stop-loss tightness | Sweep wick + 10–15 pts | Sweep wick OR 3-tick fallback |

On 1h bars, each "session" (e.g., LONDON_OPEN 02:00–05:00 UTC = 3 bars) has too few bars for reliable swing detection (requires ≥ 2*lookback+2 = 6 bars with lookback=2). The scanner fires on whatever structure forms, which may not match true ICT Silver Bullet conditions.

**Required action:** Export 1m bars from TradingView or a futures data vendor for at least MES and MGC before drawing production conclusions.
